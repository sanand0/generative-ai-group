Maya: Exactly. And the first thing that jumped out to me was shared context across agents. Sudharshan Babu kicked it off by asking, what’s the best way to let multiple agents carry context from each other, so if one model hits a limit, another can take over with less friction.

Alex: That’s such a practical question. And Sudharshan’s answer was refreshingly simple: use shared files. He said a markdown file for plans and status works very well, plus claude.md for long-term memory.

Maya: Right, and that’s the kind of advice that sounds almost too basic until you’ve tried to coordinate agents and everything falls apart. Shared files like agents.md, memory.md, and claude.md give you a durable handoff point.

Alex: Sudharshan also pointed to Rules Sync by Dhruv Anand, and even mentioned using a human project tool like Asana if you really need that level of coordination.

Maya: Then Sagar Sarkale Work added an important nuance. He said the communication layer in Bhanu’s Mission Control felt like running a company of agents, because all agents read and write to the same task threads, and @mentions help close the loop.

Alex: That’s the non-obvious takeaway: the “memory” problem is often a workflow problem. If agents can see the same task thread, same notes, same status, you get continuity without fancy orchestration.

Maya: And Sudharshan shared a really cool experiment from Bhanu: Mission Control, a custom dashboard orchestrating 10 specialized Clawdbot and OpenClaw agents to manage SiteGPT’s operations.

Alex: So for listeners building agent systems, the lesson is simple. Don’t start by inventing a magical memory layer. Start with a shared source of truth, then add automation around it.

Maya: Another strong thread was the model and tool setup people are using for coding. Sachin Kalsi said Claude Code in the VSCode extension felt slow and sometimes got stuck, but the CLI version was much better.

Alex: And he wasn’t alone. Mahesh Sathiamoorthy said the same thing: “CLI is a superior harness.” That’s a good phrase. A harness is just the surrounding setup that helps the model work better.

Maya: Mahesh also said Codex felt methodical and precise, while Claude Code felt more like an enthusiastic collaborator that can cheat or confabulate more.

Alex: That’s interesting because it suggests different roles. One tool may be better for deep, careful execution, while another is better for interactive back-and-forth.

Maya: Mahesh even said Codex had fantastic one-shotting abilities. If you throw it into a new codebase, it can keep going for 20 to 25 minutes without breaks or hallucinations.

Alex: But he also noted the Codex app itself “sucks ass,” which is a very technical phrase in the wild.

Maya: The overall point is clear though: model quality matters, but the interface and harness matter almost as much. A great model in a clunky wrapper can feel worse than a slightly weaker model in a better workflow.

Alex: And Ashish had a clever workaround. He connected Codex as an MCP tool to Claude Code and asked Claude to consult it for brainstorming and tests.

Maya: That’s a great example of using one agent to check another. Ashish said Codex is brilliant as a reviewer for Claude, while Sandeep Srinivasa said he prefers “filesystem is the API,” meaning the shared folder and skills are the interface, not some heavy messaging layer.

Alex: That really ties back to the first topic. Files, tasks, and shared notes keep showing up because they’re simple, visible, and easy to resume after a context limit.

Maya: On the frontier-model side, the group had a huge reaction to Opus 4.6 and GPT-5.3 Codex. Divyajot Singh shared that Opus 4.6 launched with a 1M context window, and said if Claude Code gets that, it will be way stronger because compaction is the biggest bottleneck.

Alex: Compaction is basically when the system has to compress older context so the conversation can keep going. A bigger context window means less of that friction.

Maya: Divyajot also pointed out that Codex seems to manage context size really well already, and that GPT-5.3 Codex is faster and more token-efficient.

Alex: There was also a strong practical message from Divyajot: these models are not a sumo match. The point is to build with them, ship with them, and stop waiting for the perfect setup.

Maya: That’s a healthy counterweight to all the hype. He said reliability is an edge in software engineering, which is a nice reminder that “best” often means “most dependable,” not just “most impressive.”

Alex: Another major area was infrastructure for reinforcement learning and agent environments. Mahesh asked about Harbor, and Sandeep explained that in their setup, browser-based rollouts make things messy because most frameworks assume quick reward loops.

Maya: He said heavy browser environments need careful sequencing so GPUs aren’t sitting idle. That matters because a lot of RL tooling assumes tiny, simple environments, but real agent work often lives in browsers, codebases, and long-running tasks.

Alex: He also called out openenv as pretty bad for their use case because the API forces synchronous responses, so they built an async pipeline in between.

Maya: That’s a great example of architecture being shaped by the work, not the other way around. If the environment is slow, your pipeline has to absorb the slowness without wasting compute.

Alex: There was a related thread on SkyRL too. Mahesh said SkyRL is in a bit of a bad shape since the main guy left to Thinking Machines, but then added that if it’s meant as an open-source tinker API, that’s kind of validation.

Maya: Which is a good reminder that in open source, founder movement doesn’t always mean failure. Sometimes it just means the idea was good enough to fork, absorb, or evolve elsewhere.

Alex: Health AI got a lot of attention too. Mohamed Yasser shared a story about Ant Afu in China, and Mahesh asked about benchmark quality for August AI and broader clinical evaluation.

Maya: Mahesh made a really important point there: medicine needs more than QA-style benchmarks. We need realistic tests that reflect clinical practice, ideally run by a fair and trusted regulator, maybe even something like ICMR.

Alex: That’s a big deal because multiple people noted that open-source MCQ datasets can be useful, but they don’t tell you whether a model behaves safely in the messy real world.

Maya: Mahesh also said governance in health AI is still being figured out, even in the US, and that some proposed assurance-lab approaches have died. So the field is moving, but slowly.

Alex: There was also a nice practical thread on local models. Sachin asked what local LLM to use for basic extraction tasks, and the suggestions ranged from LFM2-350M-Extract to Qwen 3 in various sizes.

Maya: Mohamed Yasser recommended Liquid AI’s LFM2-350M-Extract as best for that case, saying it can run on edge or CPU. That’s useful because “basic extraction” often doesn’t need a giant model.

Alex: And Sachin later said Qwen 3 4B, 8B, or 14B would probably work too, depending on how much you can run. The broader lesson is that many extraction tasks are much lighter than people assume.

Maya: Aiswarya Sankar also pointed to Entelligence CLI, which handles code review, instrumentation, and docs, and Sachin recommended Cursor CLI as well. So the CLI wave is definitely here.

Alex: The voice and multilingual side was another lively area. Aiswarya celebrated Sarvam’s Samvaad voice agents platform, saying the demo was amazing and joking about calling Clawdbot in Hindi.

Maya: That matters for product pricing and fairness. Language support isn’t just about accuracy; it’s also about how expensive it is to serve users in different languages.

Alex: Divyajot shared another cool piece: Sarvam’s Indic 3B VLM for document intelligence, with support for 22 Indic languages.

Maya: He got a very pragmatic answer: sometimes you use the customer’s cloud-native choices, and sometimes you ask them to create an AWS account under their org and hand it over, so you manage the control plane but not the data.
