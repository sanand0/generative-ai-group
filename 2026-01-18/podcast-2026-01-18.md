Maya: He also described a workflow where you start a Sprite, open Claude in it, and tell it to keep checkpointing. That’s a really useful image for non-technical listeners: the agent is no longer just a chatbot, it’s a worker in a little sandboxed room.

Maya: A VM is a virtual machine, basically a computer inside your computer. The point is isolation. If the agent does something messy, it’s contained.
Alex: That matters because agents are getting powerful enough that safety is no longer theoretical. The tool has to protect you from the tool.
Maya: Jyotirmay put it bluntly: “literally claude cli is so hyper optimised on the linux shell that it is far more productive to run a damn vm on macos...than to natively run it on macos.”
Alex: That’s funny, but it also tells you where the market is going. The best environment for the model may not be the prettiest one. It may be the safest, most controllable one.
Maya: Which ties into Cowork. Several people said Cowork feels like Claude Code for non-coding work. Jyotirmay described it as “a non terminal interface for claude code.”
Alex: So instead of asking it to edit files or run shell commands, you ask it to handle slides, tasks, research, or workflows.
Maya: That’s a big deal for knowledge workers. SARVESH GHARAT said Anthropic is quietly winning the enterprise productivity market, and that Cowork is designed to boost the personal productivity of people in enterprises.
Alex: The takeaway is simple: the interface matters less than the loop. If the model can plan, act, check itself, and store state, it becomes useful for a lot more than coding.
Maya: And there’s a business angle too. People kept saying Anthropic seems very focused on enterprise seats, workflow orchestration, and persistent business tasks.
Alex: In plain English, that means recurring company work, not one-off chats.
Maya: Right. And if those workflows get sticky, the product becomes part of how teams operate.

Alex: Another huge thread was enterprise strategy, especially around Claude Cowork, Anthropic, and how these products collect feedback.
Maya: Jyotirmay made a strong prediction: Anthropic may eventually sell enterprise seat licenses, plus serverless compute, workflow orchestration, and search indexes so people can build business workflows that run continuously.

Maya: He also said, “Anthropic has cracked the data collection and training from their harnesses loop.” A harness is the setup that tests and measures the agent.
Alex: And that’s important because good products learn from real use. The more people use the system, the more the system can improve its own workflows, if the company has the right feedback loop.
Maya: There was also a discussion about keeping rules and workflows in sync across Cursor, Claude Code, and Antigravity. Jyotirmay’s advice was very practical: narrow it down to a couple tools, and keep a central repository for rules.
Alex: That’s a great takeaway for teams. Don’t let every developer invent a new way of prompting the agent. Standardize the rules, then tweak them based on what works.
Maya: He even suggested storing those rules in things like agents.md and claude.md. That’s not flashy, but it’s the kind of thing that makes a team much faster over time.

Alex: Let’s talk about agent learning, evals, and continual improvement, because that came up a lot too.
Maya: Jyotirmay asked about the best continual learning algorithms in production for agents that get detailed feedback and rubric-based evals.
Alex: This is a super important problem. You get tons of user feedback, but how do you turn that into a better prompt, better knowledge base, or better judge model?
Maya: GEPA came up as one answer. Jyotirmay said several teams are using it in production. The basic idea is evaluator-optimizer: improve the judge, then improve the agent using that judge.
Alex: In simple terms, it’s like training the referee and the player together.
Maya: He also mentioned DSPy and ACE. ACE stands for Agentic Context Engineering. That’s basically the craft of shaping the context the model sees so it performs better.
Alex: The practical takeaway is that “feedback” isn’t enough anymore. Teams need a repeatable loop: collect feedback, update rubric, test against benchmarks, check regressions, then deploy.
Maya: Exactly. That turns agent improvement into something closer to software engineering, with versioning and test gates.

Alex: Another fascinating sub-thread was around voice agents and turn-taking.
Maya: Atharva Raykar described a hospital voice agent that fails on single-word answers and gets stuck in ugly loops.
Alex: That’s a very real product problem. Humans say “yes” or “no” or “okay” all the time, but those short answers can confuse the pipeline.
Maya: SARVESH GHARAT suggested a simple fix: use delimiters and stop generation when you hit one. That means mark where the answer ends so the system knows to stop.
Alex: He also suggested expanding short answers into longer ones, which is delightfully low-tech.
Maya: And Siddhartha Saxena brought up VAD, which is voice activity detection, basically the part that figures out when someone is speaking. Siddhartha Saxena recommended YAMNet and shared pipecat-ai/smart-turn.
Alex: The non-obvious lesson here is that many “AI failures” are not model failures. They’re pipeline failures.
Maya: Right. Sometimes the best fix is better turn detection, better prompting, or a smarter delimiter, not a bigger model.

Alex: There was also a lot of buzz about new research and tools that make long-context and stateful agents more practical.
Maya: Atharva shared AgentFS, which uses SQLite as a persistent file system. That’s a neat idea: transient compute with persistent state.
Alex: Meaning the agent can run in a temporary environment, but still remember what happened because state lives somewhere durable.
Maya: Siddhartha also shared Recursive Language Models, or RLMs. The simple version is: instead of stuffing an entire giant document into the model, let the model inspect parts of it recursively, almost like an investigator opening drawers one by one.
Alex: That matters because context windows are still limited. If you can stretch beyond them intelligently, you can handle much larger tasks.
Maya: And then there was the Anthropic Economic Index and Google’s medical models like MedGemma and MedASR. That points to the same trend: more specialized models, better edge deployment, and more use cases outside general chat.
Alex: The big idea is that AI is moving from “one giant model for everything” toward a stack of models, tools, and workflows tuned for the job.

Alex: One more thread I want to hit: the world is clearly normalizing around agent platforms, and the tooling ecosystem is consolidating.
